## 1. 先建立共同语言：核心术语（10 分钟）

### 1.1 Event
- Sentry 中最基础的一条记录
- 常见类型：Error Event、Transaction Event、Message Event

### 1.2 Transaction
- 一次完整操作的耗时过程（例如页面打开、启动）
- 在 Android 中常用于首屏、页面加载、关键用户动作

### 1.3 Span
- Transaction 内的子步骤（网络、数据库、渲染、解析等）
- 用于回答“到底慢在哪一步”

### 1.4 Trace
- 把同一次链路中的 transaction/span 串起来
- 适合跨端、跨服务定位问题

### 1.5 关系图（讲图重点）
```text
Trace
└── Transaction (一种 Event)
    ├── Span 1
    ├── Span 2
    └── Span N
```

## 2. Android 接入路径（15 分钟）

### 2.1 接入前准备
- 项目环境与版本要求
- DSN、environment、release、dist 规划
- 隐私与合规边界（PII、日志脱敏）

### 2.2 基础接入
- Gradle 插件与 SDK 依赖
- Application 初始化（最小配置）
- 采样率配置：`tracesSampleRate`、`profilesSampleRate`

### 2.3 自动采集 vs 手动埋点
- 自动能力：崩溃、部分性能链路、设备上下文
- 手动能力：自定义 transaction/span、业务标签、breadcrumb
- 什么时候必须手动：关键业务流程、首屏链路、复杂异步流程

### 2.4 上线前校验清单
- 能否看到测试异常
- 版本、环境、用户字段是否正确
- 混淆映射与符号化是否生效

## 3. 重点监控项与指标（15 分钟）

### 3.1 稳定性
- Crash Free Users / Sessions
- Java/Kotlin 异常聚合、ANR 监控
- 高频 issue 的优先级排序策略

### 3.2 性能
- 启动性能：冷启动/热启动关键耗时
- 页面性能：首屏渲染、交互响应
- 网络与 I/O：慢请求、慢查询、串行瓶颈

### 3.3 卡顿与渲染
- Slow/Frozen Frames（若版本支持）
- UI 线程阻塞与长任务识别
- 与业务场景结合的阈值定义

### 3.4 可观测性上下文
- Breadcrumb、Tag、Extra、User
- 如何让 issue 从“可见”变“可定位”

## 4. 实战演示（15-20 分钟）

### 4.1 演示 1：主动上报一个异常
- 触发测试异常并在后台定位
- 看 issue 聚合、堆栈、设备信息、版本信息

### 4.2 演示 2：手动创建首页加载 transaction
- 使用 `Sentry.startTransaction(...)`
- 给关键阶段打 span（请求、解析、渲染）
- 对比优化前后耗时

### 4.3 演示 3：用 breadcrumb 还原用户路径
- 点击、页面跳转、网络错误串联
- 从“结果异常”追溯到“触发过程”

## 5. 告警与排查闭环（10 分钟）

### 5.1 告警设计
- 告警对象：新 issue、回归 issue、阈值超标
- 告警渠道：IM/邮件/工单
- 降噪策略：采样、分组、环境隔离

### 5.2 分诊流程（建议标准化）
1. 判断影响范围（用户量、版本、机型）
2. 定位主链路（transaction + span）
3. 识别根因（代码、配置、依赖、外部服务）
4. 形成修复与回归验证

### 5.3 复盘模板
- 触发条件
- 发现路径
- 修复动作
- 预防策略（监控补点、阈值调整）

## 6. 常见坑与最佳实践（10 分钟）

### 6.1 常见坑
- 采样率过高导致成本上升
- 只上报错误不补上下文，难定位
- release/environment 命名混乱
- 混淆文件未正确上传导致堆栈不可读

### 6.2 最佳实践
- 建立统一埋点规范（命名、标签、字段）
- 关键页面必须有 transaction + 核心 span
- 与发布流程联动（版本、映射、灰度）
- 以业务指标驱动监控优先级

## 7. 团队落地路线图（5 分钟）

### 7.1 两周试点
- 1-2 个核心页面接入性能链路

### 7.3 季度治理目标
- 降低高频 issue 数量
- 提升 Crash Free Users
- 缩短 MTTR（平均修复时间）
